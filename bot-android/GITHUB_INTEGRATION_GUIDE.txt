4. AUTO-UPDATE MECHANISM (5-Minute Script Polling)
================================================================

The APK doesn't wait for user to manually update. Instead:

4.1 ScriptManager (Android Service)
-------------------------------------
Location: bot-android/android/app/src/main/java/com/supremebot/ScriptManager.kt

Behavior:
  1. On app launch:
       - Load BotPrefs.cachedScript (instant offline startup)
       - Call onUpdate() callback immediately
  2. Start infinite polling loop (Dispatchers.IO):
       - Every 5 minutes:
         a. HTTP GET https://raw.githubusercontent.com/.../candle_V12_SUPREME.js
         b. Send header: If-None-Match: <BotPrefs.scriptVersion>
         c. Check response:
            - 304 Not Modified: script unchanged, skip
            - 200 OK: new script, parse ETag header
            - If ETag != stored version:
              * Save to BotPrefs.cachedScript
              * Save to BotPrefs.scriptVersion
              * Call onUpdate(newScript, newVersion)
              * Fire onChecked() callback (for UI countdown timer)

4.2 Hot-Reload (No App Restart)
---------------------------------
When new script is detected:

  ScriptManager.onUpdate(script, version)
    -> MainActivity.onUpdate handler:
         -> BotPrefs.cachedScript = script
         -> webView.evaluateJavascript("document.readyState") { state ->
              if (state?.contains("complete")) {
                  injectScript(script)  // wrap in IIFE + eval
              }
         }

  Result: New bot logic executes IMMEDIATELY on Pocket Option page
          No app restart required
          No user action needed

4.3 Server-Initiated Reload (Optional)
---------------------------------------
Alternatively, the control server can force reload:

  POST /api/script/reload (on Node.js server)
    -> Server fetches GitHub
    -> Broadcasts to all APPs: {type:"reload_script", script, version}
    -> APK BotSocketClient receives message
    -> Calls onReloadScript() handler
    -> Same hot-reload as above

  Timeline: Instant (via WebSocket, no polling delay)

4.4 Update Status in UI
------------------------
The APK shows countdown to next update check:

  Status bar: "🔄 4:55" (4 minutes 55 seconds until next check)
  Updates every second in MainActivity.startStatusPolling()
  Formula: remaining = pollIntervalMs - (now - scriptManager.lastCheckMs)
  When remaining <= 0: shows "🔄 ..."

5. GITHUB ACTIONS WORKFLOW (build-apk.yml)
================================================================

5.1 What Triggers the Build?
-----------------------------
Events:
  ✓ Push to branch: main
  ✓ Push to branch: claude/supreme-pred-v2-engine-UiUx4
  ✓ Paths changed:
      - bot-android/android/** (any file in Android project)
      - candle_V12_SUPREME.js   (the bot script)
  ✓ workflow_dispatch (manual trigger via GitHub UI)

When triggered:
  1. GitHub spins up ubuntu-latest runner (fresh VM)
  2. Clones repository at the commit SHA
  3. Runs all steps in sequence
  4. If any step fails: workflow stops and notifies
  5. On success: creates artifact and stores for 30 days

5.2 Build Steps
----------------
1. actions/checkout@v4
     Clone the repo at the current commit

2. actions/setup-java@v4
     Install JDK 17 (Temurin distribution)
     Enable gradle caching (faster rebuilds)

3. Install Android SDK (using pre-installed $ANDROID_SDK_ROOT)
     sdkmanager "platforms;android-34" "build-tools;34.0.0"
     Takes: ~2 minutes (cached, so often skipped)

4. Create local.properties
     echo "sdk.dir=$ANDROID_SDK_ROOT" > bot-android/android/local.properties
     Gradle needs this to find Android SDK path

5. Make gradlew executable
     chmod +x bot-android/android/gradlew
     Required on Linux runners

6. Run Gradle build
     cd bot-android/android
     ./gradlew assembleDebug --no-daemon --stacktrace
     GRADLE_OPTS: -Xmx3g -XX:MaxMetaspaceSize=512m
     Duration: ~3-5 minutes (depends on cache hits)
     Output: app-debug.apk

7. Upload APK artifact
     Name: SupremeBot-debug
     Path: bot-android/android/app/build/outputs/apk/debug/app-debug.apk
     Retention: 30 days
     Accessible: From Actions tab, or via API

5.3 Critical gradle.properties Settings
-----------------------------------------
These MUST be present in bot-android/android/gradle.properties:

  android.useAndroidX=true
    → Uses AndroidX libraries (required by all modern deps)
    → Without this: "useAndroidX property is not enabled" error

  android.enableJetifier=true
    → Auto-converts old support libs to AndroidX
    → Fallback for any legacy dependencies

  org.gradle.jvmargs=-Xmx3g -XX:MaxMetaspaceSize=512m
    → Allocates 3GB heap to Gradle (for APK generation)
    → Without this: OOM errors on larger projects

  org.gradle.daemon=false
    → Disables Gradle daemon on CI (prevents hanging processes)

  kotlin.code.style=official
    → Enforces Kotlin formatter rules

14. DISASTER RECOVERY
================================================================

14.1 Script Update Fails
------------------------
If new candle_V12_SUPREME.js is broken:
  Problem: BotSocketClient crashes or spams errors
  Solution:
    1. Claude reverts commit on GitHub
    2. APK's next 5-min poll gets good script
    3. Recovery in 5 minutes max

14.2 APK Build Fails
---------------------
If GitHub Actions build breaks:
  Problem: Gradle error prevents APK generation
  Solution:
    1. Actions shows error logs (all stderr captured)
    2. Claude reads logs, identifies issue
    3. Fixes code, pushes again
    4. Next build attempt auto-triggered
    5. No user impact (old APK still works)

14.3 Server Down
-----------------
If Node.js server crashes (Railway/Render):
  Problem: APK can't connect to control server
  Solution:
    1. APK's BotSocketClient has exponential backoff (retry every 2-30s)
    2. Bot continues trading (cached script still runs)
    3. Server restarts (auto-restart on platform)
    4. APK reconnects within 1-2 minutes
    5. No trade execution lost

14.4 GitHub Outage
-------------------
If GitHub is down:
  Problem: Script polling fails temporarily
  Solution:
    1. ScriptManager retries after failure
    2. BotPrefs.cachedScript already loaded
    3. Bot continues with cached version
    4. Once GitHub recovers, APK polls again
    5. Zero impact to trading (cache is live code)
