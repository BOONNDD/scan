─── 3.1 اعتراض WebSocket (السطور 1737-1842) ───

البوت يستخدم JavaScript Proxy لاعتراض كل WebSocket جديد:

  const NativeWS = W.WebSocket;           // حفظ WebSocket الأصلي
  W.WebSocket = new Proxy(NativeWS, {
    construct(Target, args) {
      const ws = new Target(...args);      // إنشاء الاتصال الحقيقي
      _attachWSHooks(ws, String(args[0])); // تعليق الـ hooks
      return ws;
    }
  });

─── 3.2 تحديد مقبس التداول (دالة _isTradeSocket) ───

ليس كل WebSocket في الموقع هو مقبس التداول. الدالة تُمَيّز:

  function _isTradeSocket(urlStr) {
    if (urlStr.includes('po.market'))  return true;   // رابط الـ API الرئيسي
    if (urlStr.includes('chat-po'))    return false;  // مقبس الدردشة — تجاهل
    if (urlStr.includes('events-po'))  return false;  // مقبس الأحداث — تجاهل
    if (urlStr.includes('socket.io') && urlStr.includes('api')) return true; // API Socket.IO
    return false;
  }

الروابط المستهدفة تحتوي على:
  - "po.market" ← الرابط الرئيسي لـ API PocketOption
  - "socket.io" + "api" ← بديل Socket.IO

─── 3.3 تعليق الـ Hooks على المقبس (دالة _attachWSHooks) ───

عند اكتشاف مقبس التداول:

  tradeWS     = ws;           // مرجع للمقبس نفسه
  tradeWSOrig = _origSend;    // مرجع لدالة send الأصلية
                              // يُستخدم لاحقاً لإرسال أوامر التداول

  // ─ اعتراض send (الرسائل الصادرة) ─
  ws.send = function(data) {
    if (typeof data === 'string' && data.startsWith('42')) {
      // رسالة Socket.IO — راقب محتواها
      const arr = JSON.parse(data.slice(2));
      if (arr[0] === 'changeSymbol') onActiveAsset(arr[1].asset);    // تغيير الأصل
      if (arr[0] === 'saveCharts')   onPlatformTimeframe(arr[1]...); // تغيير الفريم
      if (arr[0] === 'openOrder')    isDemo = arr[1].isDemo;         // تحديث وضع Demo
    }
    return _origSend(data); // أرسل الرسالة الحقيقية بعد المراقبة
  };

  // ─ اعتراض رسائل الاستقبال ─
  ws.addEventListener('message', (e) => {
    const raw = e.data;
    if (raw === '2') { _origSend('3'); return; } // EIO4 ping → pong فوري
    if (raw instanceof ArrayBuffer || raw instanceof Blob) handleBinaryFrame(raw...);
    else if (typeof raw === 'string') handleTextMessage(raw, ws);
  });

  ws.addEventListener('open',  () => { wsConnected = true; });
  ws.addEventListener('close', () => {
    wsConnected = false;
    tradeWS = null; tradeWSOrig = null; _tradeSocketReady = false;
  });

─── 3.4 بروتوكول الاتصال: Socket.IO + Engine.IO ───

PocketOption يستخدم Socket.IO (EIO4) — بروتوكول ثنائي الطبقة:

  طبقة Engine.IO (EIO4):
    '0' → handshake (فتح الاتصال)
    '2' → ping من السيرفر (يتوقع '3' رداً)
    '3' → pong من الكلاينت
    '4' → رسالة بيانات (Socket.IO مضمّنة هنا)

  طبقة Socket.IO (فوق EIO4):
    '40' → اتصال namespace
    '42' → event (الأكثر استخداماً) = 42["eventName", {...data}]
    '43' → ack (تأكيد استلام)
    '45' → binary event (يسبق إطار binary)

  مثال رسالة حقيقية:
    42["openOrder",{"asset":"EURUSD","amount":1,"action":"call","isDemo":1,
      "requestId":12345678,"optionType":100,"time":5}]

─── 3.5 معالجة الرسائل الواردة ───

النص (handleTextMessage):
  - يبحث عن '42' في البداية (Socket.IO event)
  - يُحلّل الـ payload ويُعرّف الحدث: updateStream، tick، quote، successauth...
  - '45' + binary = إطار ثنائي مسبوق بـ event name

Binary (handleBinaryFrame):
  - يتحقق أولاً من Q1_SIG: توقيع ثنائي خاص للتيكات السريعة
  - إذا لم يتطابق → يُجرّب msgpackDecode (بروتوكول MessagePack)
  - يُعالج الأحداث: chafor (تغيير الفريم)، updateHistoryNewFast، tick...

─── 3.6 فك تشفير MessagePack (دالة msgpackDecode) ───

PocketOption يرسل بيانات السعر بتنسيق MessagePack (ثنائي مضغوط).
البوت يحتوي على محلّل MessagePack مكتوب يدوياً (لا يستخدم مكتبة خارجية):

  البايت الأول يُحدد نوع البيانات:
  ≤ 0x7f          → Integer صغير (0-127)
  0x80-0x8f       → Map (قاموس) صغير
  0x90-0x9f       → Array صغير
  0xa0-0xbf       → String صغير
  0xca            → Float32
  0xcb            → Float64  ← السعر يأتي هنا
  0xcc-0xcf       → Unsigned Integer (8/16/32/64 بت)
  0xd0-0xd3       → Signed Integer
  0xd9-0xdb       → String (8/16/32 بت للطول)
  0xdc-0xdd       → Array (16/32 بت للطول)

─── 3.7 استخراج التيك (دالة extractTickFromArray) ───

الرسالة الواردة (سواء JSON أو MessagePack) تُحوَّل لـ Array.
دالة extractTickFromArray تفهم صيغتين:

  الصيغة الأولى (nested):
    arr[0] = ["EURUSD", 1234567890, 1.08456]
    → asset = arr[0][0], ts = arr[0][1], price = arr[0][2]

  الصيغة الثانية (flat):
    arr = ["EURUSD", 1234567890, 1.08456]
    → asset = arr[0], ts = arr[1], price = arr[2]

─── 3.8 دورة حياة الاتصال الكاملة ───

  1. الصفحة تفتح → Tampermonkey يُشغّل السكريبت
  2. السكريبت يُبدّل window.WebSocket بـ Proxy
  3. الموقع يفتح WS → الـ Proxy يعترضه → _attachWSHooks
  4. الرابط يحتوي "po.market" → tradeWS = ws, tradeWSOrig = ws.send
  5. السيرفر يرسل '0' (handshake) → Socket.IO يُتم الاتصال
  6. السيرفر يرسل 42["successauth",...] → _tradeSocketReady = true
  7. كل 15 ثانية → البوت يرسل '3' (pong) للحفاظ على الاتصال
  8. السيرفر يرسل تيكات → تُمرَّر لـ onTick()

─── 3.9 المصادقة ───

لا يُنشئ البوت اتصالاً جديداً بـ PocketOption.
هو يعترض اتصال الموقع الموجود. المصادقة تحدث تلقائياً:

  - الموقع يرسل 42["auth",{"token":"..."}] عبر مقبسه الخاص
  - البوت يراقب الرد 42["successauth",...] ← يعرف أن المصادقة نجحت
  - عندها: _tradeSocketReady = true (يُفعّل التداول)

─── 4.2 توليد requestId الفريد (_nextReqId) ───

  function _nextReqId() {
    // يجمع: 20 بت من microseconds + 12 بت counter
    const perfNs   = Math.round(performance.now() * 1000) & 0xFFFFF;
    _reqIdCounter  = (_reqIdCounter + 1) & 0xFFF;
    return (perfNs * 4096 + _reqIdCounter) >>> 0;  // unsigned 32-bit
  }

  السبب: performance.now() يُعطي دقة sub-millisecond.
  ضرب × 1000 → microseconds. أخذ الـ 20 بت الأقل.
  الـ 12 بت counter يضمن uniqueness إذا بُنيت 4096 حزمة في نفس microsecond.
  النتيجة: رقم 32-بت موجب دائماً (>>> 0 يُحوّل لـ unsigned).

─── 5.5 بناء كائن الشمعة (buildCandle) ───

  function buildCandle(prices, startTime) {
    const open  = prices[0];
    const close = prices[prices.length - 1];
    const high  = Math.max(...prices);
    const low   = Math.min(...prices);

    const isBullish  = close >= open;
    const bodySize   = Math.abs(close - open);
    const totalRange = high - low;
    const upperWick  = isBullish ? (high - close) : (high - open);
    const lowerWick  = isBullish ? (open - low)   : (close - low);

    // خصائص محسوبة:
    isHammer          : lowerWick >= 2× bodySize && upperWick <= bodySize/2
    isInvertedHammer  : upperWick >= 2× bodySize && lowerWick <= bodySize/2
    isHangingMan      : هابطة + lowerWick >= 2× bodySize
    isShootingStar    : هابطة + upperWick >= 2× bodySize
    isBullishMarubozu : upperWick < 5% && lowerWick < 5% من totalRange (صاعدة)
    isBearishMarubozu : نفسه (هابطة)
    isDoji            : bodySize < 10% من totalRange
    isGravestoneDoji  : doji + upperWick > 70% من totalRange
    isDragonfly       : doji + lowerWick > 70% من totalRange
    tickCount         : prices.length  ← عدد التيكات في هذه الشمعة
  }

─── 5.6 Ring Buffer للشموع ───

  الشموع تُخزَّن في Array عادي (candleBuffers[asset]).
  الحجم محدود بـ MAX_CANDLES = 300.
  عند الامتلاء: shift() يحذف الأقدم.

  ملاحظة: حُول لاحقاً لـ Float64Array في بعض المتغيرات الداخلية:
    _PS.tickTs    : Float64Array(10)   ← timestamps آخر 10 تيكات
    _PS.vol_hist  : Float64Array(50)   ← تاريخ التذبذب
    _PS.ac_velBuf : Float64Array(50)   ← buffer الارتباط الذاتي

  لهذه الـ buffers: _rbPush() يُطبّق sliding window:
    function _rbPush(buf, nRef, val) {
      if (nRef[0] < buf.length) { buf[nRef[0]++] = val; }
      else { buf.copyWithin(0, 1); buf[cap - 1] = val; }
    }
  copyWithin(0,1) أسرع من shift() لأنه لا يُنشئ ذاكرة جديدة.

الجزء السابع: محرك التنبؤ SUPREME-PRED v2 (_predBarTick)
════════════════════════════════════════════════════════════════════════════════

يُستدعى في كل تيك قبل أي معالجة أخرى (السطر 2067).

─── 7.1 حساب السرعة والتسارع ───

  vel   = (price - lastPrice) / lastPrice * 100   // نسبة التغيير %
  accel = vel - prev_vel                           // تغيير السرعة

  يُخزَّن في ps.vels (آخر 60 قيمة)
  يُخزَّن في ps.accels (آخر 60 قيمة)

─── 7.2 تحديث EMA Stack ───

  e2  = e2_prev  + k2  × (price - e2_prev)
  e3  = e3_prev  + k3  × (price - e3_prev)
  ... وهكذا لكل الفترات الثمانية

─── 7.3 حساب Tick-RSI (Wilder Smoothed) ───

  if gain (vel > 0): tRsi_ag = tRsi_ag × 13/14 + vel/14
  if loss (vel < 0): tRsi_al = tRsi_al × 13/14 + |vel|/14
  RSI = 100 - 100/(1 + tRsi_ag/tRsi_al)

─── 7.4 حساب Z-Score (محدّث — rolling window) ───

  يستخدم آخر 30 سرعة من ps.vels:
  zn  = min(vels.length, 30)
  zm  = mean(vels[last 30])
  zstd = std(vels[last 30])
  zScore = (vel - zm) / zstd

─── 7.5 Permutation Entropy (m=3) ───

  ينظر لكل 3 سرعات متتالية → يُصنّفها في 6 أنماط ترتيبية:
  ABC: 0<1<2, 0<2<1, 1<0<2, 1<2<0, 2<0<1, 2<1<0
  H = -Σ(p × log2(p)) / log2(6)
  نتيجة عالية = فوضى | منخفضة = نظام

─── 7.6 Kaufman Efficiency Ratio ───

  العينة: آخر 15 سرعة
  ER = |Σvels| / Σ|vels|   (net / total)
  قريب من 1 = اتجاه صافٍ | قريب من 0 = فوضى

─── 7.7 2D Kalman Filter ───

  الحالة: [x (موضع), v (سرعة)]
  المعادلات:
  // Predict
  x_pred = x + v×dt
  P00 += P11×dt² + 2×P01×dt + Q_p
  P01 += P11×dt
  P11 += Q_v
  // Update
  K0  = P00 / (P00 + R)
  K1  = P01 / (P00 + R)
  innov = measurement - x_pred
  x = x_pred + K0 × innov
  v = v_prev + K1 × innov
  P11 -= K1 × P01
  P01 -= K1 × P00
  P00 -= K0 × P00

  يُعطي: kf2_vScore = v / (|v| + 0.0001)  → سكور -1..+1

─── 7.8 DFA Hurst Exponent ───

  نافذتان: n=8 و n=16 على ps.vels
  لكل نافذة:
  1. احسب cumulative sum (random walk)
  2. قسّم على windows صغيرة (4 نقاط)
  3. لكل window: ابنِ خط regression وحسب F (fluctuation)
  4. H = log(F) / log(4)  (DFA scaling)
  النتيجة النهائية: متوسط الاثنتين

─── 7.9 DFT Cycle Signal ───

  استخدام Goertzel algorithm على آخر 32 سعر:
  يُحسب للترددات: 1/8, 1/6, 1/4, 1/3 (من الطول)
  يجد أعلى magnitude → الدورة السائدة
  يُحسب الطور الحالي → BUY قرب القاع (-π/2) | SELL قرب القمة (+π/2)

─── 7.10 VWAP التيك-الموزون ───

  في كل تيك:
  vwap_psum += price × volume_tick  (حجم = |vel|)
  vwap_tsum += volume_tick
  vwap = vwap_psum / vwap_tsum
  vwapDev = (price - vwap) / vwap × 100  → انحراف %

─── 7.11 مجموعات الخوارزميات والأوزان ───

كل خوارزمية تُعطي score في [-1, +1]:
  +1 = BUY قوي جداً
  -1 = SELL قوي جداً
   0 = محايد تماماً

  المجموعة A (وزن 45%):
    ofi, vel, accel, tRsi, mom, zScore, entropy,
    ac1, ac2, ac3, ac4, ac5, tveAccel, tveBias,
    mer (Kaufman ER), ofiDelta, permEnt, vwapDev

  المجموعة B (وزن 30%):
    hurst, lr (linear regression), kalman, roc (rate of change),
    breakout, geo (candle geometry), kf2vel (2D Kalman velocity), dftCycle

  المجموعة C (وزن 15%):
    regime_w, dynSR (dynamic support/resistance), emaStack, liqVac

  المجموعة D (وزن 10%):
    candle, rsi, macd, bb (Bollinger), srsi, mtf (multi-timeframe), fib

─── 7.12 حساب النتيجة الإجمالية ───

  لكل مجموعة:
    gA = Σ(score_i × aw[i]) / Σ(aw[i])   // weighted average

  النتيجة النهائية:
    raw = gA × 0.45 + gB × 0.30 + gC × 0.15 + gD × 0.10

  تحويل لـ direction + spConf:
    direction = raw > 0.25  → 'BUY'
              = raw < -0.25 → 'SELL'
              = else        → 'NEUTRAL'
    spConf    = |raw| × 100  (0-100%)

─── 7.13 فلتر نظام السوق (Regime) ───

  النظام يُصنَّف بناءً على:
  TREND   : H > 0.60 و emaStack متجانس
  VOLATILE: ATR > 1.8× متوسط ATR الأخير
  RANGE   : ما عدا ذلك

  في نظام VOLATILE: يُشترط spConf >= 85%

─── 7.14 NEURAL-TICK: Packet Spike Detection ───

  يتتبع timestamps آخر 4 تيكات في Float64Array(10):

─── 9.5 Kelly Criterion ───

الجزء الثالث عشر: أداء النظام (PERF)
════════════════════════════════════════════════════════════════════════════════

  كائن PERF يقيس التأخيرات:
  PERF.mark('packetRecv')  → عند استقبال إطار WS
  PERF.mark('tickRecv')    → عند استخراج التيك
  PERF.mark('orderSent')   → عند إرسال الأمر

  يُحسب:
    tick latency  = tickRecv - packetRecv   (وقت فك التشفير)
    total latency = orderSent - packetRecv  (من الاستقبال للإرسال)

  Ghost Execution يُقلل total latency من ~2ms إلى < 1ms.

الجزء الرابع عشر: تدفق البيانات الكامل من بداية التيك لإرسال الأمر
════════════════════════════════════════════════════════════════════════════════

  t=0ms    : SيرVR يُرسل إطار WS (text أو binary)
  t=0.1ms  : PERF.mark('packetRecv')
  t=0.2ms  : handleTextMessage/handleBinaryFrame
  t=0.5ms  : msgpackDecode (إذا binary)
  t=0.6ms  : extractTickFromArray → { asset, price, ts }
  t=0.7ms  : PERF.mark('tickRecv')
  t=0.8ms  : onTick() → currentCandles تحديث
  t=0.9ms  : NEURAL-TICK: فحص burst
  t=1.0ms  : _predBarTick() → vel, accel, EMA, RSI, Kalman...
  t=1.5ms  : تحديث spConf و direction في _PS
  t=1.6ms  : Ghost: إذا spConf >= 55% → بنِ _ghostExecPacket
  t=2.0ms  : Signal Watcher poll (كل 100ms يتحقق)
             إذا spConf >= 70% → executeTrade()
  t=2.1ms  : Ghost Execution: packet جاهز مسبقاً
  t=2.15ms : tradeWSOrig(packet) → إرسال فوري
  t=2.2ms  : PERF.mark('orderSent')
  t=2.3ms  : UI update: log + HUD + arrows

  الفرق الحقيقي مع Ghost Execution:
    بدون Ghost: من t=2.0ms يحتاج بناء الحزمة (~1-2ms إضافية)
    مع Ghost   : الحزمة جاهزة → إرسال فوري في < 0.1ms
